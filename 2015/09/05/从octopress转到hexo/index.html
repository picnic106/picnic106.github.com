从octopress转到hexo

发表于 2015-09-05   |   分类于 hexo ， octopress   |  

概述

去年用的是octopress，今年重装操作系统，想换个生成器，所以找到了hexo，没想到还挺好用的，装的东西也比octopress要少，毕竟js的库比ruby要多，也比ruby要好用。只需安装nodejs和git就够了。而且自己也可以学学nodejs，nodejs是最近几年很火的一门语言，之前js只做前端，现在依靠谷歌的v8引擎连后端可以做。

软件安装

操作步骤:
1.安装git
2.安装nodejs
3.安装nodejs包管理器npm
4.安装hexo

安装git

git是一个开源的版本管理器，安装nodejs的npm需要用到git，npm在github开源了，另外在使用hexo部署到github也需要用到git。
安装git参考官网安装方式。
我是安装的GitHub for Windows,这个安装方式自带了一个shell，使用git打开这个shell就可以。不过还是需要掌握一些git简单命令，以及git的运行方式，这里有教程：Git教程。

安装nodejs

上nodejs官网下载双击安装即可。在命令行输入node -v测试是否安装成功。

安装npm

npm是nodejs的包管理器，很多必须的nodejs包，用这个直接安装就行，相当于java的jdk。通过git命令直接从github网站clone下来就ok，你npm已在github开源。

  3. scaffolds:模板文件。

  6. source:这个就是博客源文件放置的位置。
  7. themes:博客主题放置位置，如果我们找了一些主题或者自己新建主题，都是放在这个目录。

将octopress博文迁移到hexo

其实两个系统本质上是相同的，就是将博客生成静态文件之后部署到git服务器，所以我们只需要将octopress下的用markdown或者其他标签语言写的post文件，拷贝过来放到hexo的/hexo文件夹/source/_posts文件夹下。
但是还有几点是需要注意的地方：

  1. 因为octopress和hexo用的默认post文件名格式不一样，所以在hexo需要指定post文件名格式，这样将post文件复制到hexo，hexo才会按照这个格式来解析。修改_config.yml 中的 new_post_name 参数。

  2. octopress中的文章yaml标记和hexo还是有一些区别，例如octopress中的categories更像hexo中的tags，而hexo中的categories是有层次和序列的，和octopress中的categories不是一个概念。

  3. octopress中使用的代码高亮插件是coderay，但是在hexo中却是codeblock。
  4. 在hexo中，代表段落的#，后面需要加空格，但是octopress却不需要。
  5. octopress中如果要加目录需要使用{:toc},但是在hexo中，不需要总是这么做，有些主题会自动生成，例如我使用的next主题。

因为在octopress已经写了40多篇博客，一个一个改实在麻烦，使用python批量修改，python代码:python替换文件内容。

git介绍

其实前面这些步骤都不难，问题很有可能就处在git这一步，你必须熟悉github相关知识和命令，不熟悉者，请再次熟悉git教程，这本书也不错，一些简单命令必须要学会。

如果大家熟悉svn的话，知道svn是集中式的版本管理器，但是git是分布式的版本管理器，分布式就是说，没有所谓的服务器，每一台机器都是服务器，都有一个项目完全的拷贝，这样即使git服务器down了，还有很多拷贝，这个是学习git的先行概念。而且集中式，必须要能连上svn服务器，但是git，只要别人有git库，你就可以连上他的库。

git初始化hexo部署库

请注意：按道理，执行hexo deploy会帮助我们在这个目录初始化，但是当我在做的时候，实际上不会，所以，自己手动在hexo目录初始化库了。如果你的可以正常部署，那么这一步可以略过。

安装git之后，先要在本地也初始化一个git库，要不然如何和github服务器沟通呢。
但是这里要注意的是，因为hexo在执行hexo deploy是会将生成的静态文件copy到.deploy_git文件夹下，所以呢git库要在这个目录初始化。

之后就会在.deploy_git文件夹下新建一个.git的文件夹，这个文件夹下是git相关库文件，具体是什么我们不需要去动它。

然后新建本地分支：

> cd %nodejs-home%/.deploy_git
> git branch master #创建一个master分子和到.deploy_git文件夹

切换到master分支:

> cd %nodejs-home%/.deploy_git
> git checkout master

git add .命令将当前目录制作一个快照snapshot（快照只是登记留名，快照不等于记录在案，git管快照叫做索引index)。这一步是必须的。
然后使用git commit命令将快照里登记的内容永久写入git仓库中，也就是开发者已经想好了要提交自己的开发成果了。

> cd %nodejs-home%/.deploy_git
> git add .
> git commit -m "第一次提交" #将内容提交到本地库

这里一定要了解git是分布式的版本管理方式，所以git commit只是提交到本地库的当前分支
但是我要提交到远程github上，所以先修改.deploy_git/.git/config文件夹:

[branch "master"]
    remote = git@github.com:你的github账号/你的github账号.github.com.git
    merge = refs/heads/master

然后在命令行执行如下命令就ok了：

> cd %nodejs-home%/.deploy_git
> git push master master #将本地的master分支提交到远程的master分支上

我本机环境是，我如此执行完如上命令，hexo deploy才完全起作用，可能跟我之前用octopress部署过有关系。

git开发工作流

你的本地仓库由 git 维护的三棵“树”组成。第一个是你的工作目录，它持有实际文件；第二个是 缓存区（Index），它像个缓存区域，临时保存你的改动；最后是HEAD，指向你最近一次提交后的结果。

git遇到的问题

  1. hexo deploy报error:src refspec master does not match any
    将本地GIT版本库PUSH到一个GITHUB上一个空的版本库时可能会出现如下错误
    error:src refspec master does not match any
    原因: 本地版本库为空, 空目录不能提交 (只进行了init, 没有add和commit)
    所以我才执行如上很麻烦的操作。

  2. hexo deploy时候也没有报错，但是就是没有更新到github，如何解释？
    后来发现是在_config.yml中名/值中要有空格例如：type：（空格）git。

  3. Error: Permission denied (publickey)
    出现这个问题是没有将密钥加入到ssh-agent中，先ssh-add -l看是否存在ssh,没有再执行一次ssh-add ~/.ssh/id_rsa就好了。
